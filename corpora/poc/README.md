# PoC: detection IS reachable once the FB body is translated

This proof-of-concept de-risks the ESBMC-PLC-Sec contribution. It shows that the
**only** thing blocking detection of the Iacobelli logic bombs is that the ESBMC-PLC+
LD frontend does not translate user-defined function-block *bodies* into the GOTO IR.
When the same bomb logic is given to ESBMC directly, it is detected immediately and
its trigger is recovered.

## The bomb
`EQ_0` from `PLC-LD-dataset/malicious/mstart_eq.xml` (Iacobelli et al., ICPS 2024),
faithfully transcribed to C (same manual-harness methodology the ESBMC-PLC+ artifact
uses for its ST benchmark `motor_simple.c`):

```st
OUT := (IN1 = IN2);              (* mimics a normal EQ comparison *)
i := 0;
if IN1 = 12 then
  while i < 4 do OUT := FALSE; end_while;   (* i never increments -> infinite loop *)
end_if;
```
A non-termination / denial-of-control bomb: when the integer input equals 12 the scan
cycle hangs forever.

## Result (ESBMC v8.3.0, the ESBMC-PLC+ binary)

```
esbmc poc/eq0_legit.c     --unwind 10 --z3   ->  VERIFICATION SUCCESSFUL
esbmc poc/eq0_malicious.c --unwind 10 --z3   ->  VERIFICATION FAILED
      Violated property: unwinding assertion loop 1  (the while loop)
      Counterexample:    IN1 = 12               <-- trigger recovered automatically
```

(unwinding assertions are ON by default; `--no-unwinding-assertions` disables them.)

Three facts established:
1. **Detection** — the non-termination bomb is caught (unwinding-assertion violation).
2. **Trigger synthesis** — ESBMC recovers the exact secret value `IN1 = 12` with no
   prior knowledge.
3. **Soundness** — the benign FB verifies SAFE (no false positive).

## Why the LD pipeline misses it (proven via `--goto-functions-only`)
The malicious `.ld` declares the FB instance (`ASSIGN EQ_00=0`) but the FB *body*
(`if IN1=12 … while`) is **absent** from the GOTO program. Malicious GOTO ~= legit GOTO.
So no LD-level safety property can distinguish them.

## Generalization across all 6 trigger families (`families/`)
The bomb pattern is identical across every derived-FB family in the dataset: a real
comparison/arithmetic body plus a non-termination payload `if IN1=<trig> then while
i<4 do … end_while`. Faithful C transcriptions (`families/<FB>_{legit,malicious}.c`,
generated by `families/gen_run_families.sh`) run on ESBMC v8.3.0 give:

```
FAMILY  | LEGIT | MALICIOUS | TRIGGER (expected -> recovered)
--------+-------+-----------+--------------------------------
EQ_0    | SAFE  | VIOLATION | IN1=12 -> IN1=12
LE_0    | SAFE  | VIOLATION | IN1=12 -> IN1=12
LT_0    | SAFE  | VIOLATION | IN1=12 -> IN1=12
GE_0    | SAFE  | VIOLATION | IN1=12 -> IN1=12
GT_0    | SAFE  | VIOLATION | IN1=12 -> IN1=12
SUB_0   | SAFE  | VIOLATION | IN1=25 -> IN1=25
```
6/6 bombs detected (unwinding-assertion violation), 6/6 triggers recovered exactly,
0 false positives on the benign FBs. (EQ_0/LE_0/LT_0/GE_0/GT_0 = comparison triggers,
BOOL output, trigger 12; SUB_0 = arithmetic, INT output, trigger 25.) These 6 FB types
cover the 14 "bomb-in-FB" programs of the Iacobelli dataset.
